# Phase 1：需求分析

> 输入：`docs/01-客户资料/` 中的会议纪要
> 输出：需求分析报告 + 用户需求池 + 澄清问题清单
> 时间：Day 1-2

## 你的角色

你不在会议现场。你的输入是二手信息（纪要/录音转写），你的输出是给 FDE 的结构化产出。

## 执行步骤

### 1. 逐份阅读纪要

- 按人物提取诉求（谁说了什么）
- 标注原文位置（方便回溯）
- 区分事实陈述和主观期望

### 2. 需求提取与分类

| 类型 | 判断标准 | 举例 |
|------|---------|------|
| 明确需求 | 用户直接说"我要/我需要" | "我们需要背调报告自动生成" |
| 隐含需求 | 用户描述痛点，没明说解决方案 | "每次拜访准备花2小时"→ 效率工具 |
| 伪需求 | 可能不是真正需要的 | "要不做个区块链存证"→ 需验证 |

### 3. 去重合并

- 不同人说了同一件事 → 合并，标注所有来源人
- 不同说法描述同一需求 → 统一描述

### 4. 找遗漏

检查逻辑完整性：
- 有"录入"但没有"查询"？
- 有"生成报告"但没有"数据从哪来"？
- 多部门涉及同一数据，谁负责维护？
- 权限问题有没有提到？

### 5. 生成澄清问题

每个问题必须：
- 带 2-3 个选择项（不是开放性）
- 标明来源需求
- 标明为什么需要确认

### 6. 输出

使用模板：
- `templates/requirement-analysis-report.md` → 需求分析报告
- `templates/user-requirement-pool.csv` → 需求池
- `templates/clarification-questions.md` → 澄清问题

### 7. 主动标出不确定的点

分析完后，不要等 FDE 逐条审查。主动列出你最不确定的 3-5 个点：
- 优先级判断不准的
- 需求描述模糊的
- 逻辑上有矛盾的

## HITL 交互模式

```
AI → FDE: "分析完了，48个需求。有3个我不确定：
           1. 跨部门同步是 P0 还是 P1？
           2. 定时周报三个场景都要？
           3. 毕业衔接4月做吗？"

FDE: [审查，补充观察] "1→P0  2→只走访  3→不做。另外漏了走访周报。"

AI: [更新需求池] "已更新。等你明天带回客户反馈。"
```

## 常见陷阱

- **不要自己编造需求**：只提取纪要中有的，遗漏用"AI 发现的潜在遗漏"标注
- **不要替客户做决策**：不确定的用澄清问题标出，让 FDE 去确认
- **不要一股脑全做 P0**：P0 应该是最小可行集合
- **不要忽略"顺便提一下"**：客户随口说的往往是隐藏需求
