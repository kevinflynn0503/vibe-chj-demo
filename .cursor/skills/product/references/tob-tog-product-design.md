# ToB/ToG产品设计方法论

> ToB = To Business，面向企业客户
> ToG = To Government，面向政府/公共机构
> 核心特点：决策链长、价值驱动、深度服务

---

## 一、ToB产品核心特征

### 与ToC的本质区别

| 维度 | ToC | ToB |
|-----|-----|-----|
| **用户** | 使用者=决策者=付费者 | 三者往往分离 |
| **痛点** | 个人问题 | 组织问题 |
| **价值** | 体验、情感、效率 | ROI、成本、风险 |
| **选型** | 个人喜好 | 多人评审 |
| **销售** | 自助购买 | 销售驱动 |
| **成功** | 用得爽 | 用得好且可证明 |

### 企业产品的三类用户

```
┌──────────────────────────────────────────────────────────┐
│                      企业客户                             │
├──────────────┬──────────────┬────────────────────────────┤
│   决策者      │    影响者     │         使用者            │
│   CEO/CTO    │   部门经理    │        一线员工           │
├──────────────┼──────────────┼────────────────────────────┤
│   关心ROI    │   关心业绩    │        关心好不好用       │
│   看战略价值  │   看部门价值  │        看个人效率         │
│   决定是否买  │   推动/阻碍   │        决定用不用         │
└──────────────┴──────────────┴────────────────────────────┘
```

## 二、决策链分析

### 企业采购决策流程

```
问题识别 → 需求定义 → 方案搜索 → 供应商评估 → 选择决策 → 实施使用 → 效果评估
   │          │           │            │            │           │           │
  谁发现的？  谁定义的？   谁收集的？    谁评估的？    谁拍板的？   谁用的？    谁验收的？
```

### 决策角色地图

| 角色 | 关注点 | 如何影响 |
|-----|--------|---------|
| **发起者** | 解决自己的问题 | 让他成为内部推动者 |
| **使用者** | 好不好用、学习成本 | 降低使用门槛，提供培训 |
| **影响者** | 部门KPI、资源分配 | 证明能帮他完成目标 |
| **把关者** | 安全、合规、预算 | 满足技术和采购要求 |
| **决策者** | 战略价值、风险控制 | 高层案例、行业背书 |

### 绘制决策链的方法

```markdown
## 决策链分析模板

### 1. 列出所有参与者
- 姓名/职位
- 在决策中的角色
- 对我们的态度（支持/中立/反对）

### 2. 画出影响关系
谁影响谁？谁向谁汇报？

### 3. 找到关键节点
- 谁的一票否决会终止项目？
- 谁能推动项目加速？

### 4. 制定影响策略
针对每个关键角色，准备什么材料？传递什么信息？
```

## 三、ToB价值框架

### 企业购买的三大理由

| 价值类型 | 说明 | 证明方式 |
|---------|------|---------|
| **增加收入** | 帮企业多赚钱 | 案例：某客户使用后收入增长X% |
| **降低成本** | 帮企业省钱 | 计算：节省人力X人/年 |
| **控制风险** | 帮企业避免损失 | 场景：如果不用会有什么风险 |

### ROI计算框架

```
ROI = (收益 - 成本) / 成本 × 100%

收益 = 增加的收入 + 节省的成本 + 避免的损失
成本 = 软件费用 + 实施费用 + 培训费用 + 隐性成本
```

### 价值主张模板

```markdown
## 产品价值主张

### 一句话价值
[产品名] 帮助 [目标客户] 在 [场景] 中解决 [问题]，
实现 [可量化的收益]。

### 量化价值
- 效率提升：[X]%
- 成本节省：[X]万/年
- 收入增长：[X]%
- 风险降低：[具体场景]

### 客户证言
"使用[产品]后，我们的[指标]提升了[X]%。"
—— [客户名], [职位], [公司]
```

## 四、ToB产品架构设计

### 多租户架构

```
┌────────────────────────────────────────────────┐
│                    应用层                       │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐     │
│  │  租户A   │  │  租户B   │  │  租户C   │     │
│  └──────────┘  └──────────┘  └──────────┘     │
├────────────────────────────────────────────────┤
│                   平台层                        │
│  认证授权 │ 配置管理 │ 计费计量 │ 监控告警     │
├────────────────────────────────────────────────┤
│                   数据层                        │
│  数据隔离 │ 备份恢复 │ 数据迁移 │ 审计日志     │
└────────────────────────────────────────────────┘
```

### 数据隔离模式

| 模式 | 说明 | 适用场景 |
|-----|------|---------|
| 独立数据库 | 每个租户独立数据库 | 高安全要求 |
| 共享数据库独立Schema | 同一数据库，不同Schema | 中等隔离需求 |
| 共享数据库共享Schema | 用tenant_id区分 | 成本敏感 |

### 权限管理设计

**RBAC模型（基于角色的访问控制）**

```
用户 → 角色 → 权限 → 资源

示例：
- 管理员：所有权限
- 部门经理：本部门数据读写
- 普通员工：个人数据读写
- 访客：只读
```

## 五、客户成功体系

### 客户生命周期

```
获客 → 实施 → 上线 → 使用 → 扩展 → 续费
  │      │      │      │      │      │
售前   交付    CSM   CSM    销售   CSM
```

### 客户成功经理（CSM）职责

| 阶段 | 目标 | 关键动作 |
|-----|------|---------|
| 上线 | 顺利启用 | 培训、配置、数据迁移 |
| 早期 | 建立习惯 | 跟进使用情况、解决问题 |
| 稳定 | 深化使用 | 推广最佳实践、扩展功能 |
| 扩展 | 增购 | 识别扩展机会、协助销售 |
| 续费 | 续约 | 提前预警、价值回顾 |

### 客户健康度评分

| 维度 | 指标 | 权重 |
|-----|------|------|
| 使用活跃度 | DAU/MAU、功能覆盖率 | 30% |
| 价值实现 | 客户是否达成目标 | 25% |
| 关系健康 | NPS、沟通频率 | 20% |
| 财务状况 | 付款及时性、预算变化 | 15% |
| 扩展潜力 | 未使用功能、新需求 | 10% |

## 六、ToB定价策略

### 常见定价模式

| 模式 | 说明 | 适用场景 |
|-----|------|---------|
| **按座位** | 按用户数收费 | 协作工具（Slack、飞书） |
| **按用量** | 按使用量收费 | 基础设施（AWS、云服务） |
| **按功能** | 分版本，功能不同 | SaaS（基础版/专业版/企业版） |
| **按交易** | 按成交金额抽成 | 交易平台 |
| **一次性** | 买断制 | 传统软件 |

### 版本分层策略

| 版本 | 目标客户 | 功能范围 | 定价策略 |
|-----|---------|---------|---------|
| 免费版 | 小团队、个人 | 基础功能 | 引流 |
| 专业版 | 中小企业 | 核心功能+高级功能 | 主力收入 |
| 企业版 | 大企业 | 全功能+定制+服务 | 利润中心 |

### 定价原则
1. **基于价值而非成本**：你为客户创造多少价值？
2. **设置价格锚点**：用高价版本衬托中价版本
3. **降低尝试门槛**：提供免费试用或低门槛入门版
4. **鼓励长期承诺**：年付比月付便宜

## 七、ToG产品特殊考量

### ToG与ToB的关键差异

| 维度 | ToB | ToG |
|-----|-----|-----|
| 预算 | 商业预算，灵活 | 财政预算，周期性 |
| 采购 | 企业自主决策 | 招投标，政府采购 |
| 周期 | 周~月 | 月~年 |
| 标准 | 行业标准 | 国家/行业强制标准 |
| 安全 | 商业安全 | 等保、国产化要求 |

### ToG核心要求

#### 1. 合规性
- 等保2.0（根据系统等级）
- 信创/国产化要求
- 数据本地化存储
- 密码法合规

#### 2. 安全性
- 传输加密
- 存储加密
- 访问控制
- 审计日志

#### 3. 可追溯
- 完整操作日志
- 数据变更历史
- 谁在什么时间做了什么

#### 4. 稳定性
- 高可用设计
- 灾备方案
- 长期支持承诺

### ToG销售特点

**关键成功因素**：
- 资质齐全（等保证书、涉密资质等）
- 成功案例（同级别、同行业）
- 本地化服务能力
- 关系维护
