## 二、场景需求详析

### 场景 1：企业走访（客户拜访 app）

#### 核心诉求

蔡建总（产促部）的核心诉求是 **标准化走访流程，让新兵能用老法师的经验出去谈客户**。具体拆解为：

1. **走访前**：自动生成背调报告 + 谈话提纲（3+3 框架），减少人工准备时间
2. **走访后**：妙记自动关联 → AI 提取走访记录 → 沉淀到企业画像，消灭手工填表
3. **跨部门**：打破部门信息墙，A 部门走访的信息 B 部门也能看到
4. **管理层**：绩效看板（走访量、触达率），但顾总暂缓

### 场景 2：政策服务（政策服务 app）

#### 核心诉求

薛科总（科创中心）的核心诉求是 **扩大政策服务覆盖面 + 提高申报成功率**。KPI 直接挂钩新增高新技术企业数量。

1. **找企业**：从 17000 家园区企业中自动筛出符合高新技术企业认定条件的潜在企业
2. **做触达**：给项目经理分配走访任务，带着必问话术去确认企业申报意愿
3. **做服务**：辅助申报材料审核（前置诊断），判断材料是否符合要求
4. **看进度**：政策统计看板 + 工作进度看板

### 场景 3：孵化器管理（孵化器管理 app）

#### 核心诉求

赵婧总（孵化器）的核心诉求是 **帮孵化企业拿到更多订单**。A6 楼内约 70 家孵化企业，需要和漕河泾 17000+ 园区大企业产生商业合作。

1. **订单匹配**：大企业发需求 → 自动拆解 → 匹配孵化企业能力
2. **双向推荐**：孵化企业有变化（融资/新产品） → 自动推荐园区合作方
3. **可视化**：关系网图，展示企业间的潜在合作关系
4. **日常运营**：会议频次、访客量等运营指标监控

#### 明确需求

| UR-302 | 孵化企业新入/有变化时，推荐园区合作方 | 赵婧总 | P0 |
| UR-303 | 可交互的关系网图，展示合作可能性 | 赵婧总 | P1 |
| UR-304 | 高活跃企业识别（多会议、多访客、多招聘→可能在扩张） | 赵婧总 | P1 |

## 三、跨场景共性需求

| 共性能力 | 涉及场景 | 说明 |
|---------|---------|------|
| **统一企业数据库** | 全部 3 个场景 | 走访要查、政策要筛、孵化要匹配——底层都是同一个企业库 |
| **企业画像页** | 全部 3 个场景 | 点进一家企业能看到所有维度的信息（工商/走访/政策/合作） |
| **飞书妙记解析** | 走访 + 政策 | 走访记录生成和政策触达总结都依赖妙记内容提取 |
| **飞书任务分发** | 走访 + 政策 | 需求闭环和政策触达都要自动生成飞书任务 |
| **deep research** | 走访 + 政策 | 背调报告生成和政策初筛都需要公开信息补齐 |
| **部门知识库** | 走访 + 政策 | 赛道 3+3 话术、政策 7 条必问话术 |

> **架构建议**：上下文平台（数据层） + 企业画像（公共 UI 层） + 3 个场景 App（业务层）。企业画像不应重复建设，三个场景共享一个企业画像页。

## 四、风险与遗漏

### 已识别的遗漏

| # | 遗漏 | 说明 | 建议 |
|---|------|------|------|
| 1 | **权限管理** | 不同部门看到不同数据，蔡建总提到"有的消息不适合发给他" | 一期简单处理（管理者看全部，员工看自己部门），二期做细 |
| 2 | **数据更新频率** | 企业数据库 3 个月更新一次，走访信息实时产生 | 区分静态数据（定期同步）和动态数据（实时写入） |
| 3 | **方言和语音识别** | 蔡建总提到"万一方言太多，妙记播出效果超级差" | 走访记录生成后必须让人确认，不能直接入库 |
| 4 | **搜索和检索** | 用户会需要按企业名搜索、按标签筛选、按时间范围过滤 | 每个列表页都需要搜索和筛选能力 |
| 5 | **移动端适配** | 招商经理外出走访时需要在手机上查看 | 一期基于飞书 webview 即可，不单独做 App |
| 6 | **通知机制** | 跨部门推送、任务分发、异常预警都需要消息通知 | 统一走飞书消息 |
| 7 | **数据导出** | 0209 周会有人提到想导出数据给豆包用 | 提供关键报告的飞书文档/PDF 导出 |

### 潜在风险

| # | 风险 | 影响 | 应对 |
|---|------|------|------|
| 1 | **员工抵触录音** | 妙记数据不足，走访记录生成无法工作 | 前期作为旁路辅助，不替代现有工作流；由卫玥玥行政推动 |

## 六、AI 不确定的点

> 以下是我分析后最不确定的 5 个点，请 FDE 重点审查：

1. **跨部门信息同步的优先级**：蔡建总和薛科总都强调了这个痛点（走访信息不互通），但它涉及权限管理、通知机制等复杂功能，放 P1 是否合理？还是应该提到 P0？

2. **需求处理模块（服务匹配/合作推荐/政策推荐）的可行性**：这些功能依赖服务资源清单和产业链知识库，目前这些数据都不存在，P2 是否太乐观了？

3. **孵化器订单匹配的数据基础**：只有 10 家有详细数据、10 个历史订单案例，是否足以支撑 AI 匹配？是否应该降低承诺（先做检索展示，不做推荐）？

4. **政策触达和走访流程的关系**：政策场景的"走访"和客户拜访场景的"走访"是同一套流程吗？不同部门的走访记录汇总到同一个企业画像？这个打通一期能做到吗？

5. **Demo vs 真实可用**：0209 周会航哥提到"4 月以 demo 为主，真正用起来够呛"。但薛科总已经在期待拿去跑真实企业名单了。预期需要对齐。
