# 漕河泾驾驶舱 — 产品架构深度分析

> 日期：2026-02-10
> 作者：AI 搭档
> 背景：基于前端重构完成后，FDE 提出四个核心问题的系统性分析

---

## 问题溯源

FDE 在查看重构后的前端时，发现了四个根本性问题：

1. **企业画像是否应该单独拆出来？** — 目前三个场景各自有企业画像页，是否重复建设？
2. **缺少流程感** — 走访、政策、孵化器都是流程工作，但页面只是功能堆砌，看不出流程
3. **数据可视化缺失** — 大量统计数据、趋势数据没有图表展示，纯数字不直观
4. **功能缺失分析** — 需要系统梳理还有哪些核心功能没做

这四个问题指向一个更深层的架构问题：**当前设计是"功能视角"而非"用户旅程视角"**。

---

## 一、企业画像：应该独立，而非嵌入

### 1.1 当前问题

```
visit-app/enterprises/[id]      — 走访视角的企业画像（6个Tab）
policy-app/enterprises/[id]     — 政策视角的企业画像（政策Tab）
incubator-app/enterprises/[id]  — 孵化器视角的企业画像（合作Tab）
```

**问题**：
- 重复建设 — 基本信息、工商信息、走访记录在三个 App 里各写一遍
- 割裂体验 — 用户在走访 App 看了企业画像，切到政策 App 又是另一个页面
- 数据不一致 — 三个地方更新逻辑不同步，容易出现数据不一致

### 1.2 正确架构：企业画像是独立的"中台服务"

```
小北平台（宿主）
│
├── 🏢 企业画像 App（独立 App，公共服务）
│   ├── /enterprises                 # 企业库
│   ├── /enterprises/[id]            # 企业详情（统一入口）
│   │   ├── Tab: 基本信息            # 工商/股权/融资/团队/产品
│   │   ├── Tab: 走访记录            # 跨部门走访汇总
│   │   ├── Tab: 政策服务            # 政策筛选/触达/诊断历史
│   │   ├── Tab: 孵化合作            # 订单匹配/关系网
│   │   ├── Tab: 需求清单            # 企业诉求汇总
│   │   └── Tab: 服务记录            # 服务包/租赁/合作历史
│   └── /enterprises/[id]/report     # 背调报告（走访前准备）
│
├── 📋 客户拜访 App（场景 App，调用企业画像）
│   ├── /                            # 走访工作台（今日任务驱动）
│   ├── /records                     # 走访记录列表
│   ├── /records/[id]                # 走访记录详情
│   └── [点击企业名] → 跳转到 企业画像 App
│
├── 🛡️ 政策服务 App（场景 App，调用企业画像）
│   ├── /                            # 政策工作台
│   ├── /screening                   # 筛选任务管理
│   ├── /tasks                       # 触达任务列表
│   └── [点击企业名] → 跳转到 企业画像 App
│
└── 🚀 孵化器 App（场景 App，调用企业画像）
    ├── /                            # 孵化器概览
    ├── /match                       # 订单匹配
    └── [点击企业名] → 跳转到 企业画像 App
```

### 1.3 架构优势

| 维度 | 独立企业画像 App | 嵌入式企业画像 |
|------|----------------|--------------|
| 开发成本 | 一次开发，三个场景复用 | 三次开发，重复建设 |
| 数据一致性 | 统一入口，无不一致问题 | 三个地方同步，易出错 |
| 用户体验 | 点企业名始终跳同一个页面，一致 | 不同场景不同页面，割裂 |
| 权限管理 | 集中管理，按角色显示不同Tab | 分散管理，难以统一 |
| 维护性 | 改一处生效，易维护 | 改三处才生效，易遗漏 |

### 1.4 实施方案

**Phase 4（当前 Demo）**：
- 保持三个 App 合一的结构
- 但内部用 `<EnterpriseProfile>` 组件统一企业画像逻辑
- 企业画像路由统一为 `/enterprises/[id]`，不在各场景下重复

**Phase 5（真实部署）**：
- 拆分为独立的 `enterprise-app`
- 三个场景 App 通过 SDK 跳转到企业画像 App
- 小北平台负责 App 间导航和状态传递

## 二、流程可视化：从"功能堆砌"到"旅程导航"

### 2.1 当前问题：看不见流程

用户打开走访 App，看到的是：
```
[统计卡片] [搜索框] [表格] [侧边栏]
```

但用户的真实工作是：
```
准备走访 → 实施走访 → 确认记录 → 跟进需求
```

**问题**：页面是"功能的集合"，不是"流程的导航"。用户不知道当前在哪一步、下一步干什么。

### 2.2 三个场景的流程重构

#### 场景 1：客户拜访

**真实流程**：

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  1️⃣ 准备     │ → │  2️⃣ 走访     │ → │  3️⃣ 确认     │ → │  4️⃣ 跟进     │
│  拜访准备    │    │  实施拜访    │    │  记录确认    │    │  需求闭环    │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
  选企业              开妙记            AI提取内容         分配任务
  生成背调            录音谈话          人工确认           推送通知
  查看话术                              入库更新
```

**重构后的页面结构**：

```
/visit（走访工作台） — 流程导航中心
├── 顶部：流程步骤条（Progress Bar）
│   [准备] → [走访] → [确认] → [跟进]
│    ↑当前
│
├── 左侧：当前阶段任务卡片
│   ┌─────────────────────┐
│   │ 📋 待准备走访（3）    │
│   │ 强生医疗              │
│   │ 芯视科技              │
│   │ 蔚来汽车              │
│   │ [生成背调]            │
│   └─────────────────────┘
│
│   ┌─────────────────────┐
│   │ ⏰ 今日已安排（2）    │
│   │ 10:00 宇和科技        │
│   │ 14:00 仪电集团        │
│   │ [查看背调]            │
│   └─────────────────────┘
│
└── 右侧：待确认记录（3）
    ┌─────────────────────┐
    │ ✅ 待确认走访记录     │
    │ 2026-02-08 强生医疗   │
    │ AI 已提取，等待确认   │
    │ [立即确认]            │
    └─────────────────────┘
```

**关键改进**：
- ✅ 流程步骤条 — 用户知道当前在哪一步
- ✅ 任务卡片 — 按阶段组织，清晰驱动行动
- ✅ 快捷操作 — 一键生成背调、一键确认记录
- ✅ 时间维度 — "今日计划"比"企业列表"更符合真实工作流

### 2.3 流程导航的设计模式

| 模式 | 适用场景 | 案例 |
|------|---------|------|
| **Progress Bar** | 线性流程，每个阶段顺序执行 | 走访流程（准备→走访→确认→跟进） |
| **Funnel Chart** | 漏斗转化，每阶段人数递减 | 政策流程（筛选→触达→诊断→获批） |
| **Kanban Board** | 多任务并行，状态流转 | 需求跟进（待处理→处理中→已完成） |
| **Timeline** | 时间序列，历史追溯 | 企业走访历史、政策申报历史 |

### 3.3 可视化技术栈

| 图表类型 | 推荐库 | 说明 |
|---------|--------|------|
| 基础图表 | **Recharts** | 轻量、React 友好、适合统计类图表 |
| 复杂交互 | **Apache ECharts** | 功能强大、图表类型全、适合大屏 |
| 关系网图 | **React Flow / Cytoscape.js** | 专门用于节点关系图 |
| 地图 | **MapBox / 高德地图** | 企业地理分布 |

**推荐组合**（平衡性能和功能）：
- 基础统计图表 → **Recharts**（轻量、够用）
- 关系网图 → **React Flow**（可交互、美观）
- 大屏展示 → **ECharts**（图表类型全、效果好）

### 3.4 可视化设计规范

#### 颜色语义化

| 含义 | 颜色 | 适用场景 |
|------|------|---------|
| 成功/完成 | 绿色 `#10B981` | 已确认、已获批、高活跃 |
| 警告/待处理 | 橙色 `#F59E0B` | 待确认、待补充、中活跃 |
| 错误/异常 | 红色 `#EF4444` | 不符合、低活跃、缺失 |
| 信息/常态 | 蓝色 `#3B82F6` | A级、正常状态 |
| 次要/中性 | 灰色 `#6B7280` | 次要信息 |

## 四、功能缺失分析：系统梳理遗漏项

### 4.1 核心缺失：流程管理能力

当前实现：**数据展示** ✅  
真实需求：**流程管理** ❌

### 4.3 核心缺失：权限和数据隔离

**问题**：
- 产促部的走访记录，科创部门能看吗？
- A 项目经理能看 B 项目经理的触达任务吗？
- 孵化企业 BP 信息，谁可以查看？

**当前实现**：没有权限系统 ❌

**应该实现的权限模型**：

```
角色层级：
├── 管理者（卫玥玥、顾总） — 全局可见
├── 部门负责人（蔡建总、薛科总、赵婧总） — 本部门可见
├── 组长（赛道组长、项目组长） — 本组可见
└── 员工（招商经理、项目经理） — 自己负责的可见

数据隔离规则：
┌────────────────┬────────────┬────────────┬────────────┐
│ 数据类型        │ 管理者     │ 部门负责人  │ 员工       │
├────────────────┼────────────┼────────────┼────────────┤
│ 企业基本信息    │ ✅ 全部    │ ✅ 全部     │ ✅ 全部    │
│ 走访记录       │ ✅ 跨部门  │ ✅ 本部门   │ ❌ 仅自己  │
│ 政策筛选结果   │ ✅ 全部    │ ✅ 科创     │ ✅ 分配给自己的 │
│ 孵化企业 BP    │ ✅ 全部    │ ✅ 孵化器   │ ❌ 不可见  │
│ 统计看板       │ ✅ 全局    │ ✅ 本部门   │ ❌ 不可见  │
└────────────────┴────────────┴────────────┴────────────┘
```

**Supabase RLS 实现示例**：

```sql
-- 走访记录 RLS 策略
CREATE POLICY "员工只能看自己的走访记录"
ON visit_records FOR SELECT
USING (
  visitor_id = auth.uid()  -- 自己的记录
  OR
  auth.jwt() ->> 'role' IN ('manager', 'leader')  -- 或者是管理者
);
```

## 六、总结：从"功能"到"体验"的跃迁

### 6.1 问题根源

当前设计的本质问题：**功能视角而非用户旅程视角**

```
❌ 错误思路（功能视角）：
我有企业数据库 → 做一个企业列表页
我有走访记录 → 做一个走访记录列表页
我有筛选结果 → 做一个筛选结果页

✅ 正确思路（旅程视角）：
用户今天要走访 3 家企业 → 首页显示"今日计划"卡片
用户需要准备走访 → 一键生成背调 + 话术清单
用户走访后妙记自动提取 → 推送通知"待确认记录"
用户确认记录后提取诉求 → 自动分发飞书任务给对应部门
```

### 6.2 重构核心原则

1. **企业画像独立** — 公共服务，三个场景复用
2. **流程导航清晰** — 用户知道当前在哪一步、下一步干什么
3. **数据可视化** — 用图表代替数字堆砌
4. **任务驱动** — 首页不是列表，是"今日待办"
5. **实时通知** — 不要让用户主动查，推送通知驱动行动

### 6.3 成功指标

| 维度 | 当前 | 重构后目标 |
|------|------|----------|
| 用户理解成本 | 打开页面不知道干什么 | 3 秒内知道当前任务 |
| 任务完成路径 | 5 步才能生成背调 | 1 键生成背调 |
| 数据洞察能力 | 看数字不知好坏 | 看图表立即理解趋势 |
| 流程透明度 | 不知道走到哪一步 | Progress Bar 清晰展示 |
| 跨部门协作 | 手动通知 | 自动分发飞书任务 |
