# 角色分离与 AI-HITL 设计分析

> 日期：2026-02-10  
> 状态：分析文档，待 FDE 审查  
> 范围：visit-app 全部页面  

---

## 一、问题概述

当前 UI 存在两个结构性设计问题：

1. **角色视图混合**：老板（管理者）和员工（一线执行者）看到的是同一个界面，管理指标和执行任务混在一起
2. **手动操作假设**：所有流程都设计成「人去做」，而实际产品定位是「AI 做 + 人监督（HITL）」

这两个问题会导致：
- 员工打开页面被大量与自己无关的管理数据分散注意力
- 老板看不到全局管控所需的汇总和对比视图
- 用户以为每一步都需要手动执行，无法感知 AI 的价值
- HITL 的「确认/驳回」机制在 UI 上完全缺失

## 二、角色体系梳理

根据需求文档，存在三层角色：

### 2.1 角色定义

| 层级 | 走访场景 | 政策场景 | 孵化器场景 | 核心关注 |
|------|----------|----------|------------|----------|
| **高管** | 卫玥玥 | — | 领导层 | 全局进度、跨部门协同、汇报数据 |
| **中层管理** | 赛道组长、部门负责人 | 薛科总 | 赵婧总 | 团队进度、审核决策、规则配置 |
| **一线执行** | 招商经理 | 项目经理 | 运营人员（公台红） | 自己的任务、AI 输出的确认、数据录入 |

### 2.2 当前页面的角色归属问题

| 页面 | 当前状态 | 问题 |
|------|----------|------|
| **首页** | 全局 KPI + 三场景入口 + 实时动态 | KPI 和动态是管理视图；员工不需要全局 KPI，只需要自己的待办 |
| **政策工作台** | 转化漏斗 + 待办 + PM 工作量 | PM 工作量是**薛科总看的管理视图**；项目经理只需看自己的任务 |
| **走访看板** | 全部任务的看板 | 混合了所有人的走访任务；招商经理只该看到自己负责的 |
| **孵化器** | 运营总览 + 预警 + 企业名录 | 概况数据是赵婧总的管理视图；公台红只需看自己的待处理 |
| **走访记录** | 全部记录列表 | 管理者需要「全员走访进度」；员工只需「我的走访记录」 |
| **政策任务** | 全部任务表格 | 薛科总看所有 PM 进度；项目经理只看自己被分配的 |

### 2.3 应有的视图分离

#### 老板视图（管理驾驶舱）

> 关键词：**全局、对比、趋势、团队、审核**

| 模块 | 内容 |
|------|------|
| **全局仪表盘** | 园区企业总量、本月走访完成率、政策覆盖率、孵化活跃度 |
| **团队进度对比** | 各 PM 的任务进度（走访/政策/孵化）、转化率排名 |
| **审批队列** | 需要管理者审核的事项（诊断报告、重要推荐、异常企业处置） |
| **漏斗/管线** | 政策转化漏斗（全局视角）、走访任务完成趋势 |
| **预警看板** | 逾期任务、企业异常、数据缺失 |
| **AI 运行状况** | AI 本周处理了多少、准确率、被驳回率 |

#### 员工视图（个人工作台）

> 关键词：**我的、待办、AI 输出、确认/驳回**

| 模块 | 内容 |
|------|------|
| **我的待办** | 需要我处理的任务（AI 已准备好，等我确认） |
| **我的走访** | 今天/本周的走访安排（AI 已生成背调 & 沟通清单） |
| **AI 产出待确认** | 走访记录提取结果、需求清单、筛选结果（等我 review） |
| **我的企业** | 我负责跟进的企业列表 |
| **我的政策任务** | 分配给我的政策触达/诊断任务 |

## 三、AI-HITL 设计缺失分析

### 3.0 核心理解：HITL 是 AI 的工作流水线，人是审批节点

> **关键认知：HITL 不是"人操作、AI 辅助"，而是"AI 在跑一条完整的流水线，人只在关键检查点审批和纠偏"。**

类比走访看板：

```
走访看板（当前）= 人的工作流水线
  准备 → 走访 → 确认 → 跟进
  卡片 = 人要做的事

HITL 看板（应有）= AI 的工作流水线
  AI 采集 → AI 生成 → [人审核] → AI 分发 → [人确认] → AI 归档
  卡片 = AI 做完的事，等人过目
```

差异在于：
- **当前**：UI 展示的是人的工作步骤，按钮是"生成报告""开始筛选""发起匹配"（人驱动）
- **应有**：UI 展示的是 AI 的处理进度，按钮是"确认""驳回""修改后通过"（人监督）

这意味着整个 UI 的"主语"要从「人」切换到「AI」：

| | 当前 UI（人是主语） | HITL UI（AI 是主语） |
|---|---|---|
| 按钮文案 | "生成背调报告" | "AI 已生成，查看并确认" |
| 状态标签 | "待处理""已完成" | "AI 处理中""AI 已完成，待审核""已批准" |
| 空状态 | "暂无数据" | "AI 正在处理中，预计 5 分钟" |
| 列表含义 | "我要做的任务" | "AI 产出的结果，等我审" |
| 进度条 | "我完成了多少" | "AI 跑到哪一步了" |

### 3.1 人与 AI 的能力边界（核心）

> **原则：AI 处理信息，人处理关系和判断。AI 做得快但可能不准，人做得准但做不快。**

#### 走访场景：人与 AI 各自做什么

```
┌──────────────────────────────────────────────────────────────┐
│                       走访场景能力边界                         │
├──────────────────┬──────────────────┬────────────────────────┤
│ 环节             │ AI 做（自动）     │ 人做（不可替代）        │
├──────────────────┼──────────────────┼────────────────────────┤
│ 背调数据采集      │ 爬企查查/工商/    │ ✕ AI 全自动            │
│                  │ 新闻/专利，聚合   │                        │
│                  │ 成结构化数据      │                        │
├──────────────────┼──────────────────┼────────────────────────┤
│ 背调报告撰写      │ 基于数据生成8章   │ 审核：AI可能引用过期    │
│                  │ 节报告，提炼AI     │ 信息、理解偏差、遗漏    │
│                  │ 摘要              │ 园区内部已知信息        │
├──────────────────┼──────────────────┼────────────────────────┤
│ 沟通清单生成      │ 基于报告+赛道模   │ 审核：AI不知道上次见    │
│                  │ 板生成话术/必问/   │ 面时的人际氛围、对方    │
│                  │ 洞察              │ 性格、敏感话题          │
├──────────────────┼──────────────────┼────────────────────────┤
│ 实地走访          │ ✕ AI做不了        │ 面对面沟通、建立信任    │
│                  │                  │ 察言观色、灵活应变      │
├──────────────────┼──────────────────┼────────────────────────┤
│ 走访记录提取      │ 从妙记录音自动    │ 确认：AI可能漏听方言    │
│                  │ 转文字+提取关键   │ 误解语气、遗漏非语言    │
│                  │ 发现和诉求        │ 信息（如对方犹豫表情）  │
├──────────────────┼──────────────────┼────────────────────────┤
│ 需求分类与分发    │ 按需求类型自动    │ 确认：AI不知道部门间    │
│                  │ 匹配对口部门，    │ 的人事关系、当前工作    │
│                  │ 生成飞书任务      │ 负荷、优先级判断        │
└──────────────────┴──────────────────┴────────────────────────┘
```

**关键洞察**：整个走访场景中，只有「实地走访」是 AI 完全做不了的。其他环节 AI 都能做初稿，人的价值在于**补充 AI 不知道的隐性知识**（人际关系、现场氛围、内部信息）并**纠正 AI 可能的错误**（过期数据、理解偏差）。

#### 孵化器场景：人与 AI 各自做什么

```
│                  │                  │ 推动合作落地            │
├──────────────────┼──────────────────┼────────────────────────┤
│ 异常干预          │ ✕ AI做不了        │ 约谈企业、了解原因、   │
│                  │                  │ 提供资源支持            │
└──────────────────┴──────────────────┴────────────────────────┘
```

**关键洞察**：孵化器场景中 AI 是「千里眼」——持续监测大量数据、发现人注意不到的信号。人的价值在于**关系判断**（这个对接合不合适）和**实际行动**（安排见面、撮合合作）。

#### 总结：人 vs AI 的能力矩阵

```
AI 擅长（人做不到/做不快的）：
  - 17000家企业实时数据监控
  - 政策条件逐条批量比对
  - 录音转文字+结构化提取
  - 跨库信息聚合（企查查+走访+新闻+专利）
  - 持续24小时活跃度监测
  - 大规模需求-能力匹配

人擅长（AI做不到的）：
  - 面对面沟通和关系建立
  - 察言观色、理解弦外之音
  - 了解内部人事关系和政治
  - 判断时机是否合适
  - 在外部系统操作（政府平台/盖章/签字）
  - 处理AI没见过的异常情况

HITL的合作模式：
  AI做初稿 → 人补充隐性知识+纠正错误 → AI执行后续
  AI发现信号 → 人判断是否行动 → 人执行实际行动
  AI持续监控 → 异常时推送给人 → 人决定干预方式
```

### 3.2 三个场景的 AI 工作流水线（UI展示）

#### 走访场景 AI 流水线

```
AI 工作流（6步，2个人审节点）：

[AI] 数据采集        企查查/工商数据/新闻/走访历史 → 自动聚合
  |
[AI] 背调生成        生成 8 章节报告 + 沟通清单
  |
[人审] ★ 审核背调     招商经理确认报告 → 确认/修改/驳回
  |
[AI] 走访记录提取    从妙记录音自动提取结构化记录
  |
[人审] ★ 确认记录     招商经理确认提取结果 → 确认入库/修改/驳回
  |
[AI] 需求分发        按需求类型自动匹配部门，创建飞书任务
```

UI 应该像走访看板一样，**展示这条流水线**，让用户一眼看到每个任务跑到哪一步了：

```
+----------+    +----------+    +----------+    +----------+    +----------+    +----------+
| 数据采集  |    | 报告生成  |    | ★待审核   |    | 记录提取  |    | ★待确认   |    | 需求分发  |
| AI处理中  | -> | AI处理中  | -> | 3份待审   | -> | AI处理中  | -> | 1份待确   | -> | 自动完成  |
| 2个任务   |    | 1个任务   |    | 人工节点  |    | 0个任务   |    | 人工节点  |    | 5个已发   |
+----------+    +----------+    +----------+    +----------+    +----------+    +----------+
```

#### 政策场景 AI 流水线

```
[AI] 企业筛选          批量拆条件 + deep research → A/B/C 评级
  |
[人审] ★ 审核评级       管理者确认 A 级名单 → 通过/降级/排除
  |
[AI] 自动分发          按 PM 专长和工作量 → 生成分配方案
  |
[人审] ★ 确认分配       管理者确认分配 → 通过/手动调整
  |
[AI] 申报诊断          自动评分 + 材料清单 → 生成诊断报告
  |
[人审] ★ 审核诊断       PM 审核报告 → 通过/退回补充/驳回
```

#### 孵化器场景 AI 流水线

```
[AI] 活跃度监测         持续监控行为数据 → 发现异常/变化信号
  |
[AI] 自动匹配/推荐      需求匹配 or 反向推荐
  |
[人审] ★ 审核推荐        运营人员 → 采纳对接/忽略/调整
  |
[AI] 推送双方           自动通知匹配双方
```

### 3.3 UI 应该如何展示 AI 流程

> **核心思路：UI 展示的是 AI 的工作流水线，人的操作点用醒目的检查点标记。**

类比制造业的流水线看板：
- 流水线上的每个工位 = AI 的每个处理步骤
- 质检点 = 需要人审核确认的节点
- 看板展示的是「产品（任务）在流水线上走到了哪里」

#### 走访场景的 UI 展示

当前的走访看板按「人的工作步骤」分列：准备 -> 走访 -> 确认 -> 跟进
应改为按「AI+人的协作流程」分列：

```
当前看板（人的视角）：
  准备（人要去看资料） -> 走访（人要去跑） -> 确认（人要去确认） -> 跟进（人要去跟进）

HITL看板（AI的流程 + 人的检查点）：
  AI采集中 -> AI已生成报告 -> [人]审核报告 -> [人]实地走访 -> AI提取记录 -> [人]确认记录 -> AI分发需求
  
  其中：
  - 灰色步骤 = AI正在处理，人不用管
  - 蓝色步骤 = AI已完成，等人查看
  - 红色步骤 = 必须人来做（如实地走访）
  - 绿色步骤 = 已通过人审核
```

每张卡片上标注：
- 这一步是谁在做（AI图标 / 人图标）
- 如果是AI做的，展示置信度和数据来源
- 如果需要人审核，展示「确认/修改/驳回」按钮
- 如果是人必须做的事（如实地走访），展示日程安排

#### 政策场景的 UI 展示

当前的转化漏斗展示的是「数据结果」（17000 -> 326 -> 18 -> 8 -> 0）
应改为展示「AI处理流程的当前状态」：

```
当前漏斗（静态数据）：
  全部企业 17,000 -> 已筛选 326 -> 已触达 18 -> 有意愿 8 -> 已获批 0

HITL流程（动态状态）：
  [AI] 筛选中    -> [人]审核评级  -> [AI]生成分配  -> [人]确认分配  -> [人]走访触达  -> [AI]诊断评分  -> [人]审核诊断
  处理到15000/17000  待审核 12家A级    已生成方案      待确认 3个方案    进行中 18家      已完成 8家       待审核 3份
```

#### 孵化器场景的 UI 展示

当前展示的是「静态数据 + 操作入口」
应改为展示「AI持续监测的实时状态」：

```
当前：
  概况数据（在孵数/高活跃/匹配数） + 预警列表 + 匹配按钮

HITL视图：
  [AI] 活跃度监测         持续运行中，2分钟前更新
  [AI] 匹配引擎          1个匹配任务进行中
  [AI] 变化信号检测       今日发现3条信号
  
  → 待您处理：
    - 2条匹配结果待审核 [查看]
    - 1条异常预警待处理 [查看]
    - 3条反向推荐待确认 [查看]
```

### 3.4 当前 UI 距离 HITL 缺什么

#### 缺失 A：没有 AI 工作流视图（最关键）

当前没有任何地方展示"AI 跑到哪一步了"。用户打开页面看到的是静态数据，不知道背后有 AI 在持续工作。

**需要**：每个场景的主页面应有一个类似看板的"AI 流水线视图"，展示各步骤的任务数量和状态。

#### 缺失 B：没有 AI 生成标识

用户无法区分数据是 AI 产出还是人工填写。

**需要**：AI 产出内容统一标记 `AI 生成` + 置信度 + 数据来源。

#### 缺失 C：没有审批操作

所有页面都是"展示"，缺少核心的审批交互：
- 走访记录确认页：有查看，缺"确认入库/驳回"
- 筛选结果：有列表，缺"通过/降级"
- 匹配推荐：有结果，缺"采纳/忽略"
- 诊断报告：有展示，缺"审核通过/退回"

**需要**：每个人审节点（★）都有统一的操作按钮组。

#### 缺失 D：按钮语义是"人发起"而非"人审批"

| 当前按钮 | 传达的含义 | 应该改为 | 传达的含义 |
|---------|-----------|---------|-----------|
| "生成背调" | 人去触发 AI | "AI 已生成，待审核" | AI 已完成，等你看 |
| "开始新一轮筛选" | 人去启动流程 | "查看最新筛选结果 (3)" | AI 自己在跑，有新结果了 |
| "发起订单匹配" | 人去触发匹配 | "AI 推荐了 2 条匹配" | AI 发现了机会，等你审 |
| "分发任务" | 人手动分发 | "AI 建议分配方案，待确认" | AI 已分好，你确认就行 |

### 3.5 需求文档中的 7 个 HITL 节点实现情况

| # | HITL 节点 | AI 做什么 | 人做什么 | UI 现状 | 缺失 |
|---|----------|----------|----------|---------|------|
| 1 | **走访记录确认** | 从妙记 AI 提取 | 确认/调整/补充 | 有确认页 | 缺 AI 标识和审批按钮 |
| 2 | **需求清单确认** | AI 提取诉求整理 | 确认后分发 | 部分体现 | 缺独立确认交互 |
| 3 | **政策初筛审核** | AI 筛选+评级 | 管理者审核评级 | 有列表 | 缺置信度和审核操作 |
| 4 | **政策走访问题** | AI 生成走访问题 | PM 按问题询问 | 沟通清单体现 | 未标明哪条是 AI 基于哪个缺失生成 |
| 5 | **申报诊断审核** | AI 评分+诊断 | PM 审核结论 | 有诊断页 | 缺"通过/退回"操作 |
| 6 | **订单匹配审核** | AI 匹配 | 运营审核 | 有匹配页 | 缺"采纳/忽略"操作 |
| 7 | **反向推荐审核** | AI 推荐 | 运营审核 | 有推荐页 | 缺"采纳/忽略"操作 |

## 四、方案自我批评

在给出最终方案前，先审视初版设计的问题：

### 4.0 初版设计的三个缺陷

**缺陷1：角色切换太生硬**

初版方案用导航栏下拉切换「管理视角/员工视角」——这是一种"两个 App 硬捏在一起"的做法。问题：
- 管理者也需要做具体事务（比如赵婧总也会亲自确认某个推荐），切过去切过来很烦
- 员工偶尔也想看团队整体情况（比如看看自己在团队中排第几），不应该完全屏蔽
- 二元切换暗示了"你是老板或员工"，但实际很多人两种角色兼有

**缺陷2：AI 流水线视图太复杂**

初版画了一个 6 列的 AI 看板（数据采集→报告生成→待审核→记录提取→待确认→需求分发），这对用户来说太复杂了。员工并不关心"AI正在采集数据"或"AI正在生成报告"——他们**只关心"有什么需要我处理的"**。完整的 AI 流水线视图也许适合给产品经理做 demo 演示，但不适合日常使用。

**缺陷3：新增组件太多**

初版设计了 6 个新组件（AIStatusBar、ReviewCard、ReviewQueue、AIConfidenceBadge、HumanOverride、AutoPilotToggle），这是过度设计。更好的方式是用最少的新增元素，融入现有页面。

## 五、改进方案（v2）

### 5.0 设计哲学

> **产品名叫「智能驾驶舱」——驾驶舱不是让你开车，而是让你监控自动驾驶**

三个核心原则：
1. **收件箱模式**：不是"人去找任务"，而是"AI完成的事情推到你面前，你审完就走"
2. **渐进式角色**：不搞二元切换，而是"默认看自己的，可展开看全局"
3. **嵌入式 AI 标识**：不搞独立的 AI 面板，而是在现有页面上标注"这是 AI 做的"

### 5.1 角色分离：渐进式而非二元切换

#### 放弃「角色切换器」，改用「视图范围」

```
┌─────────────────────────────────────────────────┐
│  漕河泾智能驾驶舱                                │
│  首页  客户拜访  政策服务  孵化器  企业库         │
│                                    薛坤 · 科创中心│
└─────────────────────────────────────────────────┘

  ┌──────────────────────┐
  │ 👤 我的   👥 团队全局  │   ← 不是"角色切换"，而是"视图范围"
  └──────────────────────┘
```

差异：
- 不叫"管理视角/员工视角"，叫"我的/团队全局"——心智模型不同
- "我的"是默认，"团队全局"不需要权限，但管理者看到的数据更多（员工看团队全局只能看到脱敏的汇总，看不到其他人的具体任务）
- 每个页面可以独立切换，不是全局切换（比如走访看"我的"，同时政策看"团队全局"）

#### 「我的」视图（默认）

```
首页变成「我的收件箱」：

  早上好，薛坤                              2026年2月10日

  ┌─ 待处理 ─────────────────────────────────────────┐
  │                                                   │
  │  ✦ AI 生成了「蔚来汽车」背调报告     2分钟前      │
  │    8章节 · 5必问 · 置信度 85%                     │
  │    [查看并审核]                                    │
  │                                                   │
  │  ✦ AI 提取了走访记录（强生医疗）     30分钟前      │
  │    3条关键发现 · 2条诉求                          │
  │    [确认记录]                                      │
  │                                                   │
  │  ✦ AI 推荐了匹配：芯视科技↔蔚来      1小时前      │
  │    匹配度 95% · 传感器供应链                      │
  │    [审核推荐]                                      │
  │                                                   │
  │  📋 明天 14:00 走访蔚来汽车                        │
  │    背调已就绪 ✓ · 沟通清单已生成 ✓                 │
  │    [查看准备材料]                                   │
  │                                                   │
  └───────────────────────────────────────────────────┘

  ┌──────┐ ┌──────┐ ┌──────┐
  │待审核 │ │我的走访│ │ 转化率 │
  │  3   │ │ 6/8  │ │ 50%  │
  └──────┘ └──────┘ └──────┘
```

**关键设计**：首页不再是 KPI 仪表盘，而是一个**按时间排序的待处理队列**。这就是 HITL 的自然表达——AI 产出的东西排在这里等你过目。

#### 「团队全局」视图

```
首页变成「管理仪表盘」：

  漕河泾智能驾驶舱 · 团队全局              2026年2月10日

  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
  │企业总数│ │走访完成│ │政策覆盖│ │在孵活跃│
  │17,000│ │ 78%  │ │ 326  │ │ 78%  │
  └──────┘ └──────┘ └──────┘ └──────┘

  ┌─ AI 今日工作报告 ─────────────────────────────────┐
  │  今日自动完成 12 项 · 待团队审核 5 项 · 驳回率 8%  │
  │  背调报告 3份 · 政策筛选 1轮 · 匹配 2次           │
  └───────────────────────────────────────────────────┘

  ┌─ 团队效率 ────────────────────────────────────────┐
  │  薛坤   ████████░░ 6/8   转化50%                  │
  │  PM-A   █████░░░░░ 5/12  转化40%                  │
  │  PM-B   ████░░░░░░ 4/10  转化50%                  │
  └───────────────────────────────────────────────────┘

  ┌─ 需要关注 ────────────────────────────────────────┐
  │  🔴 芯视科技活跃度下降至35                         │
  │  🟡 蔚来汽车缺少研发费用数据                       │
  │  🟡 PM-A 有3个逾期任务                             │
  └───────────────────────────────────────────────────┘
```

### 5.2 HITL：嵌入现有页面，而非独立面板

#### 核心：不新建 AI 看板，改造现有页面

初版方案试图新建一个"AI流水线视图"——这太重了。更好的做法是**在现有的每个页面上嵌入 AI 状态和审批操作**：

| 现有页面 | 嵌入什么 | 怎么嵌入 |
|---------|---------|---------|
| 走访看板 | AI 准备状态 | 每张准备阶段卡片标注"✦ AI已生成背调"而非"待生成背调" |
| 走访确认页 | 审核操作 | 底部增加 [确认入库] [需要修改] 操作栏，标注"✦ AI 提取" |
| 政策漏斗 | AI 处理进度 | 每个阶段标注"AI处理中 15000/17000"或"AI已完成，待审核 12家" |
| 筛选列表 | 审核操作 | 每行增加 [通过] [降级] 操作，标注置信度 |
| 诊断页 | 审核操作 | 底部增加 [审核通过] [退回补充] 操作栏 |
| 匹配结果 | 审核操作 | 每条结果增加 [采纳对接] [忽略] 操作 |
| 反向推荐 | 审核操作 | 每条推荐增加 [发起对接] [暂不对接] 操作 |

#### 统一的 AI 标识规范

只需要 **2 个新增元素**，而非 6 个组件：

**元素1：AI 生成标签**（嵌入现有卡片/页面）

```
✦ AI 生成 · 10分钟前 · 置信度 85% · 来源: 企查查+走访记录
```

设计：
- `✦` 用品牌色 #3370FF
- 置信度用三级颜色：≥80% 绿 / 50-80% 蓝 / <50% 橙
- 嵌入位置：标题行右侧或内容区顶部
- 尺寸：text-[10px]，不喧宾夺主

**元素2：HITL 操作栏**（嵌入详情页底部）

```
┌──────────────────────────────────────────────────┐
│  ✦ AI 已完成处理，等待您的确认                     │
│                                                    │
│  [✓ 确认通过]    [✎ 修改后确认]    [✕ 驳回]       │
└──────────────────────────────────────────────────┘
```

设计：
- 固定在详情页底部（sticky bottom）
- 三个按钮：确认（主按钮蓝）/ 修改（次按钮灰）/ 驳回（文字红）
- 驳回时弹出原因选择

#### 导航栏 AI 状态指示

不做独立面板，只在导航栏加一个**小徽标**：

```
首页  客户拜访  政策服务  孵化器  企业库   [✦ 3]  薛坤
                                            ↑
                                     待审核数量（点击展开简要列表）
```

### 5.3 各页面具体改造

#### 首页改造

```
「我的」模式 → 收件箱
  - 去掉全局 KPI
  - 顶部显示"待处理 N 项"
  - 按时间倒序排列：AI产出（待审核）+ 日程提醒 + 任务进展
  - 底部3个个人统计卡片

「团队全局」模式 → 管理仪表盘
  - 保留全局 KPI（增加同比）
  - 增加 AI 工作报告摘要
  - 增加团队效率对比
  - 增加需要关注事项
```

#### 走访看板改造

```
「我的」模式：
  - 四列看板不变
  - 只显示我负责的企业
  - 准备阶段卡片标注 "✦ 背调已就绪" 或 "✦ AI 生成中..."
  - 确认阶段卡片标注 "✦ AI 已提取，待确认"

「团队全局」模式：
  - 显示所有人的任务
  - 顶部增加按人员分组的进度条
  - 每张卡片标注负责人
```

#### 政策工作台改造

```
「我的」模式：
  - 漏斗改为"我的任务进度"（只看分配给我的企业）
  - 去掉 PM 工作量
  - 待办只显示我的

「团队全局」模式：
  - 保留完整漏斗
  - 漏斗每个阶段标注 AI 状态（"AI筛选中 15000/17000"）
  - 保留 PM 工作量
  - 增加 AI 准确率指标
```

#### 孵化器改造

```
「我的」模式：
  - 直接显示"待我处理"列表（推荐/匹配/异常）
  - 每条标注 AI 生成信息

「团队全局」模式：
  - 保留概况统计
  - 增加 AI 监测状态
```

### 5.4 为什么这个方案更好

| 维度 | 初版方案 | v2 方案 |
|------|---------|---------|
| 角色分离 | 二元切换（生硬） | 视图范围切换（渐进） |
| HITL 展示 | 独立 AI 看板（复杂） | 嵌入现有页面（自然） |
| 新增组件 | 6 个新组件 | 2 个新元素 + 首页收件箱 |
| 用户心智 | "我要切换身份" | "我要看我的/看全局" |
| 首页定位 | KPI 仪表盘（谁都不满足） | 收件箱（员工）/ 仪表盘（管理） |
| AI 感知 | 独立的 AI 面板 | 融入每个页面的标注 |
| 改造成本 | 新建管理驾驶舱 4 个页面 | 改造现有页面 + 首页收件箱 |

## 七、不改会怎样
