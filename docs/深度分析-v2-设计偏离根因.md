## 二、本质诊断：三个根因

### 根因 1：把「工作流工具」做成了「数据展板」

**这是最核心的问题。**

产品设计文档清楚定义了三个场景，每个都是**完整的工作流**：

```
客户拜访：选企业 → 生成背调 → 带着问题去走访 → 妙记提取记录 → 确认/调整 → 需求分发
政策服务：批量筛选 → 分级 → 分发触达任务 → 跟踪走访 → 收集意愿 → 材料诊断
孵化器：  描述需求 → AI拆解匹配 → 查看结果 → 推进对接 → 跟踪进展
```

但当前实现变成了：

```
所有页面 = 只读数据展示
首页 = 数据大屏（KPI + 漏斗 + 排行）
功能页 = 列表 + 卡片（看完就走）
用户感受 = "很酷但不知道干什么"
```

**偏离的表现：**

| 产品设计 | 当前实现 | 差距 |
|---------|---------|------|
| 客户拜访首页 = 企业列表 | 首页 = 数据大屏 | 首页就偏了 |
| "生成背调"是核心操作 | 按钮存在但无流程 | 无 Agent 交互 |
| 走访记录可编辑、可确认 | 走访记录只能看 | 缺关键交互 |
| 政策筛选有"开始筛选"流程 | 政策页直接显示结果 | 没有操作入口 |
| 触达任务可分发到飞书 | 没有分发按钮 | 缺飞书联动 |
| 订单匹配是对话式 AI | 输入框无法真的发送 | 假交互 |
| 需求可以派发给其他部门 | 需求只有列表展示 | 没有闭环 |

**根因：** 开发过程中聚焦于"视觉效果优化"，把产品需求中的操作流程简化为了只读展示。每个页面都在回答"数据长什么样"，但没有回答**"用户来了之后做什么"**。

---

### 根因 2：架构决策违背了嵌入式应用的本质

产品架构明确定义了：

```
┌──────────────────────────────────────┐
│           小北驾驶舱平台                │
│  ┌───────────┐ ┌──────────┐ ┌──────┐ │
│  │ 客户拜访App │ │ 政策服务App │ │ 孵化器 │ │
│  └───────────┘ └──────────┘ └──────┘ │
│  ┌──────────────────────────────────┐ │
│  │       公共企业画像模块             │ │
│  └──────────────────────────────────┘ │
└──────────────────────────────────────┘
```

**设计意图：**
- 每个 App 是独立的，嵌入小北的一个 Tab
- 小北已经有导航，App 不需要自己的导航
- 用户在小北里切换 App，不是在 App 里切换功能
- SDK Context 提供 `sendChat` 与小北通信

**我们做了什么：**
- 把三个 App 合并成一个单体应用
- 加了自己的顶部导航栏（等于在小北的导航里又套了一层导航）
- 首页是全局 Dashboard（三个场景的数据混在一起）
- 企业画像不是公共模块而是 visit-app 的子页面

**结果：**
- 嵌入小北时，顶部导航和小北的 Tab 冲突
- 用户在小北里打开"客户拜访"，看到的却是包含政策、孵化器的全局看板
- 架构复杂度增加，三个场景互相干扰

**正确做法：** 三个 App 保持独立（或至少逻辑独立），每个 App 不带全局导航，通过 `?embedded=true` 判断是否隐藏。在 Demo 阶段可以有一个简单的入口页，但不应该是合并的单体应用。

### 根因 3：视觉设计脱离了目标用户的认知习惯

**目标用户是谁：**
- 政府开发区的招商经理、科创项目经理、孵化器运营
- 日常使用飞书、企查查、企业数据库
- 审美偏好：简洁、专业、高信息密度、浅色系
- 不是科技公司的 SaaS 用户，不是 C 端消费者

**我们做了什么：**
- 深色主题（dashboard 风格）——像 Grafana 监控台
- 大量渐变色、发光效果、动画——像 SaaS 营销页
- 圆角 2xl、超大字号、稀疏布局——浪费信息密度
- 字体用 Inter + DM Sans——国际风格，不是中文生态

**用户期望的参照物：**

| 参照物 | 特征 | 为什么合适 |
|--------|------|-----------|
| 飞书（Lark） | 浅色、紧凑、功能密集、灰蓝配色、左侧边栏 | 用户每天用 8 小时 |
| 企查查 | 浅色、信息密集、Tab 清晰、数据卡片朴实 | 企业画像的行业标杆 |
| 钉钉工作台 | 应用入口卡片 + 功能列表 | 类似的企业工具定位 |

**我们缺的不是"更好看"，是"更对味"。** 中国 B 端 / G 端（政府端）用户的审美是：
- 白色/浅灰背景
- 蓝色为主色调（信任感）
- 紧凑布局（一屏看到更多信息）
- 表格/列表为主（不是卡片网格）
- 操作按钮明确（不是隐含在卡片 hover 里）
- 状态标签清晰（颜色编码 + 文字）

## 四、产品设计 vs 当前实现——逐场景对比

### 场景 1：客户拜访

**产品设计的用户旅程：**

```
招商经理打开 App
    │
    ▼
看到企业列表（我负责的/最近走访的/未走访的）← 首页就是企业列表
    │
    ▼ 点击"强生医疗"
企业画像页（多 Tab：基本信息/走访记录/需求/服务）← 像企查查
    │
    ▼ 点击"生成背调"
背调报告页（左：报告内容 / 右：沟通清单 + 必问问题）← 有操作
    │
    ▼ 带着报告去走访，开飞书妙记
走访结束，妙记自动提取内容
    │
    ▼
走访记录出现在列表中（"待确认"状态）← 需要用户操作
    │
    ▼ 点击查看 → 编辑/补充 → 确认入库
走访记录归档，企业诉求自动进入需求清单
    │
    ▼
需求派发给对应部门（飞书任务）← 闭环
```

**当前实现的用户旅程：**

```
用户打开 App
    │
    ▼
看到深色数据大屏（KPI + 漏斗 + 排行）← ？？我是来看大屏的吗
    │
    ▼ 点击"客户拜访"Tab
看到4个指标卡片 + 走访记录列表（只读）← 没有操作
    │
    ▼ 点击记录，跳转到企业详情页
看到企业基本信息（只读）← 跳走了，回不来
    │
    ▼
...到此结束，没有后续流程
```

**差距：**
- 首页定位错了（Dashboard vs 企业列表）
- 缺少"生成背调"的完整流程
- 走访记录不可编辑、不可确认
- 没有需求分发
- 没有飞书妙记/日历/任务的联动占位

### 场景 2：政策服务

**产品设计：**

```
首页 = 政策工作台：
  - 概览卡片（本轮筛选数/A级/B级/C级）
  - 触达进度条（分发→走访→意愿→诊断）
  - 快捷操作：[开始新一轮筛选] [查看待处理任务] [看统计]
  - 最近动态流
```

**当前实现：**

```
政策页 = 数据展示：
  - 统计卡片（同上）
  - 漏斗图（只是图）
  - PM 进度表（只是看）
  - 筛选结果列表（只能点进去看详情）
```

**差距：** "开始筛选"按钮存在但没有流程。"一键分发"按钮存在但不做事。没有触达任务管理。整个页面是报告，不是工作台。

### 场景 3：孵化器管理

**产品设计：** 订单匹配页是核心——对话式 AI，左边对话右边结果卡片。

**当前实现：** 聊天界面存在但输入无效。匹配卡片有数据但不可操作。缺少"查看关系图"跳转。缺少企业画像入口。

## 五、正确方向

### 5.1 设计原则修正

| 原则 | 错误做法 | 正确做法 |
|------|---------|---------|
| 风格 | 深色科技风 Dashboard | 飞书风格：浅色、紧凑、专业 |
| 首页 | 全局数据大屏 | 每个 App 有自己的首页（列表/工作台） |
| 交互 | 展示为主 | 操作为主（每个页面都有"下一步"） |
| 架构 | 合并成单体 + 顶部导航 | 三个独立 App，无全局导航，适配嵌入 |
| 企业画像 | 简单详情页 | 企查查式：多数据源、多 Tab、信息密集 |
| 飞书生态 | 完全缺失 | 占位体现（妙记、日历、任务、文档图标+提示） |

### 5.2 视觉风格修正

```
❌ 不要：
- 深色背景
- 大面积渐变色
- 发光/毛玻璃效果
- 超大圆角（2xl）
- 稀疏布局
- Inter/DM Sans 字体

✅ 要：
- 白色/F7F8FA 浅灰背景
- 蓝色系主色（#1677FF 或 #3370FF 飞书蓝）
- 紧凑间距（py-2 而非 py-5）
- 表格/列表为主体
- 清晰的操作按钮（不是隐含在 hover 里）
- 系统字体（PingFang SC / Microsoft YaHei）
- 12-14px 为主字号
- 标准边框和分割线（不是渐变边框）
```

### 5.3 架构修正

```
方案 A：Demo 阶段简易方案（推荐）
─────────────────────────────
保持 visit-app 作为开发载体，但：
1. 去掉顶部导航栏
2. 用 URL 路由区分三个"App"
   /visit/*  → 客户拜访
   /policy/* → 政策服务
   /incubator/* → 孵化器
3. 每个路由组有自己的首页和流程
4. ?embedded=true 时完全没有壳
5. 独立访问时顶部只有当前 App 的名字

方案 B：恢复三个独立 App
─────────────────────────────
1. visit-app、policy-app、incubator-app 各自独立
2. 共享 /lib/schema.ts 和 /lib/mock-data.ts
3. 各自部署到不同端口
4. 小北中作为三个独立 Tab
```

### 5.5 飞书生态占位

即使 Demo 阶段没有真实飞书 API，也应该在 UI 中**占位体现**：

| 位置 | 占位方式 |
|------|---------|
| 走访记录 | "来源：飞书妙记 2026-01-20" + 妙记图标 |
| 背调报告 | "已同步至飞书文档" + "在飞书中打开"链接 |
| 触达任务 | "已创建飞书任务" + 任务状态 |
| 企业画像 | "数据来源：企业数据库 / 企查查 / 走访记录" |
| 日程关联 | "走访日程：2026-02-12 14:00 强生医疗" |
| 通知 | "已推送飞书消息给薛坤" |

这些占位让用户**感知到系统是飞书生态的一部分**，而不是一个孤立的数据展板。

## 六、总结

**一句话总结这些问题的本质：**

> 我们把一个**帮招商经理/项目经理做事的工作流工具**，做成了一个**给领导看的数据展板**。
>
> 视觉上追求"酷炫"而忽略了用户的认知习惯；架构上追求"统一"而违背了嵌入式应用的逻辑；功能上追求"展示完整"而忽略了操作流程的完整性。

**三句话概括修正方向：**

1. **从 Dashboard 回到 Workflow** — 每个页面都要回答"用户来了做什么"，不是"数据长什么样"
2. **从合并回到独立** — 三个 App 保持独立路由，去掉全局导航，适配嵌入
3. **从科技风回到飞书风** — 浅色、紧凑、操作明确、飞书生态占位
