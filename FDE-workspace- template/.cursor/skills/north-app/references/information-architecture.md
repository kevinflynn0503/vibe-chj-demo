# 信息架构设计

> 好的信息架构是隐形的 — 用户自然而然找到想要的，无需思考

---

## 一、信息架构核心目标

### 用户体验三阶段

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   阶段1: 认知 (Cognition)                                       │
│   ─────────────────────                                         │
│   用户问: "这是什么？能做什么？"                                 │
│   设计目标: 清晰传达产品价值和能力                               │
│                                                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   阶段2: 寻找 (Finding)                                         │
│   ─────────────────────                                         │
│   用户问: "我要的功能在哪里？"                                   │
│   设计目标: 高效的导航和搜索                                     │
│                                                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   阶段3: 使用 (Using)                                           │
│   ─────────────────────                                         │
│   用户问: "怎么完成我的任务？"                                   │
│   设计目标: 流畅的任务流程                                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 信息架构的四大系统

| 系统 | 定义 | 关键问题 |
|-----|------|---------|
| **组织系统** | 信息如何分类和结构化 | 用户如何理解这些分类？ |
| **标签系统** | 信息如何命名 | 这个词用户能理解吗？ |
| **导航系统** | 用户如何浏览和移动 | 用户知道自己在哪、能去哪吗？ |
| **搜索系统** | 用户如何精确查找 | 搜索结果符合用户期望吗？ |

## 二、组织系统设计

### 常见组织模式

```yaml
1. 层级结构 (Hierarchy):
  描述: "树状结构，从顶层到底层"
  适用: "大多数内容型产品"
  示例: 
    - 电商分类（服装 > 女装 > 连衣裙）
    - 文档导航（指南 > 快速开始 > 安装）
  
2. 矩阵结构 (Matrix):
  描述: "多维度交叉，用户可从不同入口进入"
  适用: "复杂查找需求"
  示例:
    - 筛选系统（按品牌 × 价格 × 颜色）
    - 数据表格（按列排序、筛选）

3. 线性结构 (Sequential):
  描述: "步骤式，有明确的顺序"
  适用: "流程型任务"
  示例:
    - 注册流程
    - 结账流程
    - 引导教程

4. 标签结构 (Tagging):
  描述: "扁平化，通过标签关联"
  适用: "用户生成内容"
  示例:
    - 博客文章标签
    - 收藏夹分类
```

### 分类设计方法

#### 卡片分类法

```markdown
## 卡片分类步骤

### 1. 准备阶段
- 列出所有需要分类的项目（功能、内容）
- 写成卡片（每项一张）
- 准备空白卡片用于用户自定义分类

### 2. 执行阶段
- 邀请5-8名目标用户
- 请用户将卡片分组
- 让用户为每组命名

### 3. 分析阶段
- 统计相同分组的频率
- 找出用户命名的共性
- 形成最终分类方案

### 分类类型
- **开放式**: 用户自由分组和命名
- **封闭式**: 预设分类，用户只分组
- **混合式**: 预设分类但允许新增
```

#### 分类原则

```yaml
MECE原则:
  M (Mutually): 相互独立
  E (Exclusive): 没有重叠
  C (Collectively): 完全穷尽
  E (Exhaustive): 没有遗漏

实践建议:
  - 同级分类数量: 5-9个（神奇数字7±2）
  - 层级深度: 不超过3-4层
  - 命名: 用户语言而非业务术语
  - 覆盖: 确保每个项目都有归属
```

## 三、标签系统设计

### 标签命名原则

```yaml
清晰性:
  ✅ "设置" 
  ❌ "Configuration Parameters"
  
一致性:
  ✅ "新建项目" / "新建文件夹"
  ❌ "新建项目" / "添加文件夹"
  
简洁性:
  ✅ "保存"
  ❌ "保存当前更改"
  
熟悉性:
  ✅ "购物车"
  ❌ "待购清单"
```

### 标签类型

| 类型 | 定义 | 示例 |
|-----|------|------|
| **导航标签** | 导航项的命名 | 首页、产品、关于我们 |
| **标题标签** | 页面/区块标题 | 热门商品、最新动态 |
| **链接标签** | 可点击文字 | 查看更多、了解详情 |
| **图标标签** | 图标的辅助文字 | 搜索、设置、通知 |

### 标签测试方法

```markdown
## 首点测试 (First Click Test)

### 目的
验证用户能否通过标签找到目标功能

### 执行
1. 给用户一个任务："找到修改密码的地方"
2. 记录用户第一次点击的位置
3. 统计正确率

### 标准
- 首点正确率 > 80% = 标签清晰
- 首点正确率 < 50% = 需要优化

## 5秒测试

### 目的
验证标签是否能快速被理解

### 执行
1. 展示界面5秒
2. 隐藏界面
3. 询问用户看到了什么功能

### 分析
- 用户提到的功能是否与设计意图一致
- 用户是否遗漏重要功能
```

## 四、导航系统设计

### 导航类型

```yaml
全局导航:
  定义: "始终存在的主导航"
  位置: "顶部栏/侧边栏"
  内容: "产品核心功能入口"
  示例: "首页、项目、设置、个人中心"

局部导航:
  定义: "当前区域内的子导航"
  位置: "页面内/侧边"
  内容: "当前模块的细分功能"
  示例: "设置 > 账户、安全、通知"

上下文导航:
  定义: "与当前内容相关的导航"
  位置: "内容附近"
  内容: "相关内容、推荐内容"
  示例: "相关文章、你可能还喜欢"

辅助导航:
  定义: "补充性的导航方式"
  类型: 
    - "面包屑: 显示位置层级"
    - "站点地图: 全局概览"
    - "索引: 字母/分类索引"
```

### 导航模式选择

```
┌─────────────────────────────────────────────────────────────────┐
│                         功能数量                                 │
│                    少(5-7)    中(8-15)    多(15+)               │
├──────────┬────────────────────────────────────────────────────── │
│          │                                                      │
│  简单    │  顶部导航    顶部导航      顶部+二级菜单              │
│  任务    │  (tabs)     (dropdown)                               │
│          │                                                      │
├──────────┼──────────────────────────────────────────────────────│
│          │                                                      │
│  复杂    │  侧边导航    侧边导航      侧边+收缩面板              │
│  任务    │  (简洁)     (分组)                                   │
│          │                                                      │
└──────────┴──────────────────────────────────────────────────────┘
```

### 导航最佳实践

```yaml
位置清晰:
  - 当前位置高亮
  - 面包屑导航
  - URL反映层级

可预测性:
  - 导航项位置固定
  - 交互方式一致
  - 避免隐藏重要入口

高效性:
  - 常用功能快捷访问
  - 支持键盘导航
  - 搜索作为补充

响应式:
  - 移动端汉堡菜单
  - 平板端收缩侧边栏
  - 桌面端完整展示
```

## 五、搜索系统设计

### 搜索类型

| 类型 | 适用场景 | 示例 |
|-----|---------|------|
| **全局搜索** | 全站内容检索 | 顶部搜索框 |
| **局部搜索** | 特定区域检索 | 表格内搜索 |
| **筛选搜索** | 条件组合 | 商品筛选 |
| **即时搜索** | 边输入边显示 | 命令面板 |

### 搜索体验优化

```yaml
输入阶段:
  - 自动补全建议
  - 搜索历史记录
  - 热门搜索推荐
  - 输入即搜索（防抖）

结果阶段:
  - 结果分类展示
  - 关键词高亮
  - 相关度排序
  - 无结果时的建议

优化方向:
  - 支持模糊搜索
  - 同义词支持
  - 拼写纠错
  - 搜索范围选择
```

### 搜索UI设计

```markdown
## 搜索框设计

### 位置
- 全局搜索: 顶部居中或右侧
- 局部搜索: 内容区域上方

### 尺寸
- 宽度: 根据常见搜索词长度（通常200-400px）
- 高度: 与按钮一致（通常36-44px）

### 状态
- 默认: 灰色边框，placeholder提示
- 聚焦: 主色边框，放大图标
- 输入中: 显示清除按钮
- 搜索中: 加载指示器

### 快捷键
- Cmd/Ctrl + K: 打开搜索
- Esc: 关闭搜索
- ↑↓: 选择建议
- Enter: 执行搜索
```

## 六、页面布局架构

### 常见页面类型

```yaml
列表页:
  结构: "筛选/搜索 + 列表/网格 + 分页"
  关键: "快速扫描、批量操作、排序筛选"
  示例: "商品列表、文章列表、用户列表"

详情页:
  结构: "标题区 + 内容区 + 操作区"
  关键: "信息层级清晰、操作便捷"
  示例: "商品详情、文章详情、用户资料"

表单页:
  结构: "分组表单 + 验证 + 提交"
  关键: "逻辑分组、即时验证、明确提示"
  示例: "注册表单、设置页面、编辑页面"

仪表板:
  结构: "概览卡片 + 图表 + 快捷操作"
  关键: "信息密度适中、重点突出"
  示例: "数据分析、管理后台首页"

空状态:
  结构: "图示 + 说明 + 行动引导"
  关键: "友好提示、引导下一步"
  示例: "空列表、无搜索结果"
```

### 页面布局原则

```yaml
F型阅读模式:
  描述: "用户习惯从左上到右，再向下扫描"
  应用:
    - 重要信息放左上
    - 操作按钮放右侧
    - 次要信息放下方

Z型阅读模式:
  描述: "用户视线呈Z形移动"
  应用:
    - 适用于落地页
    - 左上logo，右上导航
    - 中间内容，右下CTA

视觉层级:
  L1: "标题、主要CTA - 最大最醒目"
  L2: "副标题、次要操作 - 中等"
  L3: "正文、辅助信息 - 正常"
  L4: "注释、元数据 - 最小最浅"
```

## 七、工具产品特殊考量

### 工具产品IA特点

```yaml
效率优先:
  - 任务流程最短
  - 常用功能快捷访问
  - 支持批量操作
  - 键盘快捷键

信息密度:
  - 比消费产品更高
  - 但需要合理的留白
  - 支持密度切换

专业术语:
  - 可以使用行业术语
  - 但需要一致性
  - 提供帮助文档

高级功能:
  - 渐进式披露
  - 新手隐藏高级功能
  - 支持自定义
```

### 聚合型工具设计

```yaml
模块化设计:
  原则: "功能按属性分类，独立但可组合"
  实现:
    - 每个模块有独立的边界
    - 模块间通过明确的接口连接
    - 支持单独使用也支持组合

标准化设计:
  原则: "统一规范对接多个模块"
  实现:
    - 统一的操作模式（增删改查）
    - 统一的视觉语言
    - 统一的交互反馈

特性化设计:
  原则: "在标准基础上满足差异需求"
  实现:
    - 核心流程保持一致
    - 特定场景允许定制
    - 高级用户可自定义
```

## 八、信息架构检查清单

### 组织系统
- [ ] 分类是否符合用户心智模型？
- [ ] 同级分类是否相互独立？
- [ ] 层级深度是否合理（≤3层）？
- [ ] 每个项目是否有明确归属？

### 标签系统
- [ ] 标签命名是否使用用户语言？
- [ ] 同类标签风格是否一致？
- [ ] 标签是否简洁清晰？
- [ ] 是否做过首点测试？

### 导航系统
- [ ] 用户是否知道当前位置？
- [ ] 导航是否覆盖核心功能？
- [ ] 常用功能是否便捷访问？
- [ ] 移动端导航是否友好？

### 搜索系统
- [ ] 搜索入口是否明显？
- [ ] 是否支持自动补全？
- [ ] 无结果时是否有引导？
- [ ] 结果是否按相关度排序？
